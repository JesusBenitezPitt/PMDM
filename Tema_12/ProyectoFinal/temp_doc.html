1. Descripción del juego

Maze Runner es un videojuego 2D para Android inspirado en el clásico Pac-Man. El jugador controla un personaje que recorre un laberinto generado de forma procedural mediante un algoritmo DFS, recogiendo monedas y power-ups mientras esquiva tres tipos de enemigos con comportamientos distintos. El objetivo es llegar a la salida con la mayor puntuación posible sin perder todas las vidas.

Sonido

El juego incluye música de fondo y efectos de sonido para recoger monedas, activar power-ups, colisionar con enemigos y completar o fallar un nivel.

Controles

El juego se maneja con un D-pad táctil en la mitad inferior de la pantalla:

  • Botón ▲ – Mover el personaje hacia arriba
  • Botón ▼ – Mover el personaje hacia abajo
  • Botón ◀ – Mover el personaje hacia la izquierda
  • Botón ▶ – Mover el personaje hacia la derecha

Los botones funcionan como buffer de dirección: la dirección pulsada se aplica en cuanto el personaje termina el movimiento actual, lo que permite anticipar los giros sin que el personaje se detenga.

Cómo jugar

  1. Pulsar Jugar en la pantalla de inicio.
  2. Usar el D-pad para mover al personaje por los pasillos del laberinto.
  3. Recoger la mayor cantidad posible de monedas y power-ups.
  4. Evitar el contacto con los enemigos. Tener el escudo activo absorbe un golpe sin perder vida.
  5. Llegar a la celda verde (salida) para completar el nivel y pasar al siguiente.
  6. La dificultad aumenta con cada nivel: los enemigos son progresivamente más rápidos.

3. Desafío técnico: sistema de sprites animados

El mayor reto del proyecto fue conseguir que los sprites del personaje y los enemigos se renderizaran correctamente desde un sprite sheet compartido. El proceso pasó por tres errores encadenados que hubo que diagnosticar y resolver uno a uno.

¿Qué es un sprite sheet y cómo funciona en el juego?

Un sprite sheet es una única imagen que contiene todos los frames de animación organizados en una cuadrícula. En lugar de cargar un bitmap por frame, se carga una sola imagen (walk_sheet.png) y en cada dibujado se recorta el rectángulo exacto correspondiente al frame y dirección actuales. El jugador y los tres enemigos comparten el mismo sheet; la diferencia visual entre ellos se consigue aplicando un filtro de color (PorterDuff.Mode.SRC_ATOP) sobre el sprite base.

Dimensiones reales del sprite sheet:

PropiedadValor
Dimensiones de la imagen192 × 128 px
Columnas (frames de animación)8
Filas (direcciones de movimiento)4
Tamaño de cada frame24 × 32 px
FPS de la animación8 fps

Problema 1 – Constantes de cuadrícula incorrectas (sprite con sangrado lateral)

Las constantes estaban declaradas como WALK_COLS = 6 y WALK_ROWS = 4. Con 6 columnas el código calculaba un ancho de frame de 192 ÷ 6 = 32 px, cuando el frame real mide 24 px. Esto hacía que cada rectángulo de recorte capturase 1,33 frames del sheet, mostrando el sprite actual más una franja del siguiente. El efecto visual era que la animación “se desplazaba hacia la izquierda” al avanzar de frame.

Diagnóstico: medir la imagen con la API de Windows (System.Drawing) confirmó 192 × 128 px. La división 192 ÷ 6 = 32 no daba un número redondo de frames, pero 192 ÷ 8 = 24 sí. La imagen tenía 8 columnas, no 6.

Solución:

// Player.java y Enemy.java — antes (incorrecto)
private static final int WALK_COLS = 6;  // frameW = 32 px (captura 1,33 sprites)
private static final int WALK_ROWS = 4;

// Player.java y Enemy.java — después (correcto)
private static final int WALK_COLS        = 8;   // frameW = 24 px exacto
private static final int WALK_ROWS        = 4;   // frameH = 32 px exacto
private static final int WALK_ANIM_FRAMES = 8;   // ciclo completo de 8 frames

// Recorte del frame correcto en draw()
int frameW = walkSheet.getWidth()  / WALK_COLS;  // 192 / 8 = 24 px
int frameH = walkSheet.getHeight() / WALK_ROWS;  // 128 / 4 = 32 px
int srcX   = frameAnim * frameW;
int srcY   = row       * frameH;
Rect src = new Rect(srcX, srcY, srcX + frameW, srcY + frameH);

Problema 2 – Texture bleeding por filtrado bilineal

Aunque el tamaño de frame fuera correcto, al escalar el sprite al tamaño de celda del laberinto Android aplicaba interpolación bilineal (activada por el flag FILTER_BITMAP_FLAG del Paint). Esa interpolación mezcla los píxeles del borde del recorte con los del frame adyacente en el sheet, produciendo un “fantasma” semitransparente en el borde del sprite. Este efecto se llama texture bleeding.

Solución: eliminar el flag para forzar interpolación nearest-neighbor, que es la adecuada para sprites pixel-art:

// Antes (causaba bleeding)
private static final Paint PINCEL = new Paint(
    Paint.ANTI_ALIAS_FLAG | Paint.FILTER_BITMAP_FLAG);

// Después (nearest-neighbor, sin sangrado entre frames)
private static final Paint PINCEL = new Paint(Paint.ANTI_ALIAS_FLAG);

Problema 3 – Mapping de dirección a fila del sheet incorrecto

El switch que traducía la dirección de movimiento a la fila del sprite sheet tenía el orden equivocado respecto al orden real del archivo: pulsar izquierda mostraba el sprite mirando arriba, pulsar arriba mostraba el sprite mirando a la derecha, etc.

Solución: verificar visualmente el orden de filas del sheet e invertir el mapping:

// Antes (mapping equivocado)
switch (dirActual) {
    case DIR_ABAJO:     row = 0; break;
    case DIR_IZQUIERDA: row = 1; break;  // Fila 1 es Arriba, no Izquierda
    case DIR_ARRIBA:    row = 3; break;  // Fila 3 es Derecha, no Arriba
    default:            row = 2; break;  // Fila 2 es Izquierda, no Derecha
}

// Después (mapping correcto)
switch (dirActual) {
    case DIR_ABAJO:     row = 0; break;
    case DIR_ARRIBA:    row = 1; break;
    case DIR_IZQUIERDA: row = 2; break;
    default:            row = 3; break;  // DIR_DERECHA
}

Lección aprendida

Antes de escribir ninguna constante de sprite sheet hay que medir la imagen real: ancho ÷ columnas y alto ÷ filas deben dar números enteros exactos. Si no cuadran, las constantes están mal. Para sprites pixel-art hay que desactivar siempre el filtrado bilineal de Android (FILTER_BITMAP_FLAG) para evitar el sangrado entre frames. Y el orden de filas del sheet hay que verificarlo abriendo la imagen, no asumirlo por el nombre de las variables.
